# Участие в разработке Termif

Termif принимает изменения, которые усиливают продуктовую цель, архитектурную связность и надежность. Репозиторий не ведется как набор случайных фич: каждое изменение должно улучшать стабильность выполнения, UX, поддерживаемость границ подсистем или качество релизов.

## Контракт вклада

Pull request должен сначала объяснять продуктовый смысл, а уже потом детали реализации. Ревьюеру важно понимать, какое поведение изменилось для пользователя, какие предположения появились и какие ошибки были учтены.

Rust остается runtime-источником истины, React - слоем взаимодействия. Перенос persistence или критичной runtime-логики в компоненты считается архитектурным регрессом, даже если локально все работает.

## Границы

Frontend-изменения должны оставаться декларативными и управляемыми состоянием. Сложное поведение лучше держать в store actions или backend-контрактах, а не в случайных component effects.

Backend-изменения должны сохранять типизированные модели, явную передачу ошибок и предсказуемые команды. Ошибка должна доходить до UI в actionable виде, а не превращаться в анонимный failure.

## Проверки качества

Основной gate - CI: lint, tests, build, Rust fmt/clippy/tests. UX-изменения желательно сопровождать заметками о keyboard path, потому что Termif ориентирован на клавиатурную работу.

## Документация

Если меняются модели, runtime-поведение, команды, настройки или persistence, обновляйте документацию в том же PR. Отложенная документация быстро создает расхождение между кодом и ожиданиями.

## Безопасность

Не логируйте пароли, ключи, секреты или чувствительные SSH payloads. Пути, работающие с SSH-секретами, должны минимизировать хранение и исключать случайную сериализацию в debug output.

Плагинный runtime пока не является открытой поверхностью. Не добавляйте обходные механизмы динамического выполнения без отдельного security review.

## Готовность PR

PR готов к ревью, когда scope понятен, границы архитектуры сохранены, ошибки обработаны явно, а документация отражает итоговую семантику. Небольшие сфокусированные PR предпочтительнее широких смешанных рефакторингов.
