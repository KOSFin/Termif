## Направление

Termif развивается из desktop MVP в надежное кроссплатформенное терминальное рабочее пространство. Ближайший цикл сфокусирован не на количестве новых функций, а на доверии, стабильности SSH, предсказуемом файловом workflow и качестве релизов.

## Текущая база

В продукте уже есть кастомная оболочка приложения, локальные и SSH-вкладки, файловый менеджер по контексту активной вкладки, группы SSH-хостов, палитра команд, настройки, горячие клавиши, встроенный редактор, сохранение состояния workspace и CI-сборки для Windows, macOS и Linux.

Последние изменения усиливают эту базу: добавлена автоматическая адаптация под системную светлую/темную тему, настройка native-прозрачности macOS, лендинг GitHub Pages с загрузками из GitHub Releases, проверка SSH host key через `known_hosts`, компактный список сниппетов, адаптивные панели со скрытым скроллбаром и ограниченное восстановление terminal scrollback по вкладкам после перезапуска приложения.

## Ближайшие приоритеты

### 1. Доверие SSH-подключений

Runtime уже должен проверять или записывать host key. Следующий шаг - отдельный UI подтверждения fingerprint, понятная диагностика при смене ключа и видимость политики доверия для пользователя.

### 2. Надежность сессий

Нужно дальше стабилизировать reconnect, очистку устаревших session id, timeout-поведение и пробуждение после сна/нестабильной сети. Цель - без долгих непрозрачных ожиданий и без скрытых reconnect-циклов. Текущее восстановление scrollback помогает посмотреть прошлый вывод после рестарта, но не является восстановлением живого процесса.

### 3. Файлы и редактор

Файловый менеджер и редактор уже работают локально и по SSH, но требуют лучшей обработки больших файлов, сетевой задержки, конфликтов сохранения и ошибок remote write.

### 4. Релизы

Артефакты собираются для Windows, macOS и Linux, а лендинг умеет показывать актуальные файлы релизов. Дальше нужны подпись, notarization, стабильная политика checksum-файлов и более качественные release notes.

## Среднесрочные цели

Структурированный input overlay остается экспериментом и должен включаться только опционально.

Долговременное хранение transcript остается отдельной задачей: текущие per-tab логи в localStorage ограничены по размеру и подходят для recall, но не заменяют полноценный журнал с поиском, экспортом, retention policy и настройками приватности.

Плагинная система остается стратегическим направлением, но только после permission-модели, capability prompts и изоляции сбоев.

Платформенный UX должен сохранять единый язык, но уважать соглашения ОС: Command на macOS, Ctrl на Windows/Linux, POSIX defaults на Unix, native-прозрачность и системные темы там, где платформа это поддерживает.

## Критерии следующей крупной вехи

Веха считается закрытой, когда SSH trust flow прозрачен для пользователя, reconnect стабилен на реальных сетевых сбоях, файловые операции имеют детерминированное восстановление после ошибок, а релизы Windows/macOS/Linux имеют более сильные гарантии происхождения.
